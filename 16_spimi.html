What Is SPIMI?

Single-Pass In-Memory Indexing (SPIMI) builds an inverted index efficiently for large document collections when the full index cannot fit in memory.

1. Why SPIMI Was Introduced

BSBI generates all (termID, docID) pairs, sorts them, then builds postings lists. Sorting large numbers of pairs is expensive.

SPIMI removes the global sorting step by building postings lists directly in memory.

2. Core Idea of SPIMI

Instead of (term, docID) → sort → group, SPIMI directly does:

term → append docID to postings list

Then it writes the dictionary block to disk once memory is full.

3. Step-by-Step Algorithm

Step 1: Initialize an Empty Dictionary

Dictionary
Dictionary = {}

Each entry stores:

term → postings list

Step 2: Parse Documents Sequentially

For each token:

  • If term not in dictionary, create an entry.
  • Append docID to the postings list.

Step 3: When Memory Is Full

  • Sort dictionary terms (not pairs).
  • Write the block's inverted index to disk.
  • Clear memory and continue.

Step 4: Final Merge

Merge partial inverted indexes similarly to BSBI.

4. Why It Is Called "Single-Pass"

  • Each token is processed exactly once.
  • No global sorting of term-doc pairs.
  • Postings lists are built directly during parsing.

6. Why SPIMI Is Faster

In BSBI:

O(T log T)

due to sorting all pairs.

In SPIMI:

O(T)

for hash insertions, with only term sorting per block (smaller).

8. Advantages of SPIMI

  • Faster than BSBI.
  • No global sorting of term-doc pairs.
  • Direct postings construction.
  • Simpler implementation.
  • Better cache performance.

9. Limitations

  • Still requires a final merge.
  • Memory must be managed carefully.
  • Hash table resizing can be costly.

10. Key Insight

BSBI = "Sort first, then group"

SPIMI = "Group directly, no sorting of pairs"

11. Exam Definition

Single-Pass In-Memory Indexing (SPIMI) is an inverted index construction algorithm that processes documents sequentially, builds postings lists directly in memory using a hash-based dictionary without sorting term–document pairs, writes sorted term blocks to disk when memory is full, and merges them into a final index.

12. Worked Example (SPIMI)

Assume we have 4 documents and memory can hold only 2 documents at a time, so we build 2 blocks.

Document Collection
Doc1: "cat dog cat"
Doc2: "dog fish"
Doc3: "cat fish bird"
Doc4: "dog bird"

Block 1 (Doc1, Doc2)

After Doc1:

Dictionary
cat → [1]
dog → [1]

After Doc2:

Dictionary
cat  → [1]
dog  → [1, 2]
fish → [2]

Memory full → sort terms and write Block 1:

Block 1 (sorted terms)
cat  → [1]
dog  → [1, 2]
fish → [2]

Block 2 (Doc3, Doc4)

After Doc3:

Dictionary
cat  → [3]
fish → [3]
bird → [3]

After Doc4:

Dictionary
cat  → [3]
fish → [3]
bird → [3, 4]
dog  → [4]

Memory full → sort terms and write Block 2:

Block 2 (sorted terms)
bird → [3, 4]
cat  → [3]
dog  → [4]
fish → [3]

Final Merge

Merged Inverted Index
bird → [3, 4]
cat  → [1, 3]
dog  → [1, 2, 4]
fish → [2, 3]

Notice that we never created or sorted (term, docID) pairs. Postings were built directly.
