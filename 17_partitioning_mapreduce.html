Overview

These three ideas focus on scaling search systems to massive collections by distributing indexing and query processing across many machines.

1. Document-Partitioned Index

Idea

Split documents across machines. Each machine builds an inverted index only for its assigned documents.

How It Works

Machine Documents
M1Doc 1–250M
M2Doc 250M–500M
M3Doc 500M–750M
M4Doc 750M–1B

Each machine stores term → postings list only for its documents.

Query Processing

Architecture
Query
   ↓
Broadcast to all shards
   ↓
Each shard returns top-k
   ↓
Merge results
   ↓
Final ranking

Advantages

  • Parallel query processing.
  • Good load balancing.
  • Easy to scale by adding machines.
  • Most common design in web search.

Used In

Modern search engines such as Google and Bing.

2. Term-Partitioned Index

Idea

Split the vocabulary across machines. Each machine handles a subset of terms.

Machine Terms
M1a–f
M2g–m
M3n–z

Problem

To compute intersections like intersection(cat, dog), postings are on different machines, which causes heavy network traffic and complex merging.

  • High network communication.
  • Slow query processing.
  • Ranking becomes difficult.

Term partitioning is rarely used in production systems.

4. MapReduce for Index Construction

MapReduce is a distributed programming model introduced by Google to process huge datasets.

Map → Shuffle → Reduce

5. MapReduce for Building an Inverted Index

Map Phase

Input: documents. Emit (term, docID) for each term.

Mapper example
Doc1: "cat dog"
(cat, 1)
(dog, 1)

Shuffle Phase

Groups pairs by term:

Grouped
cat → [1, 3, 7]
dog → [1, 2, 4]

Reduce Phase

Reducers build postings lists:

Reducer output
term → sorted docIDs

6. Why MapReduce Is Powerful

  • Automatically parallel.
  • Fault tolerant.
  • Handles huge datasets.
  • Works across thousands of machines.

7. Relationship Between These Concepts

  • Document-partitioned index: used at query time.
  • MapReduce: used at index construction time.
  • Term-partitioning: alternative approach, rarely used.

8. Full Big Picture

Index construction
Documents → Map → Shuffle → Reduce → Inverted Index
Query processing
Query → Broadcast → Local scoring → Merge

9. Intuition

  • Document partitioning: splitting books across libraries.
  • Term partitioning: splitting dictionary alphabet across libraries.
  • MapReduce: factory system assembling index pieces in parallel.

10. Exam-Ready Definitions

Document-Partitioned Index: an index distributed across machines where each machine stores postings lists for a subset of documents, enabling parallel query processing.

Term-Partitioned Index: an index distributed by dividing vocabulary across machines, where each machine stores postings for a subset of terms.

MapReduce: a distributed computing model consisting of Map and Reduce phases used to build large-scale inverted indexes in parallel.

11. Worked Example: MapReduce Indexing

Assume we have three documents:

Documents

Doc2 (ID=2): "dog fish"
Doc3 (ID=3): "cat fish"

Map Phase

Map outputs
(cat, 1)
(dog, 1)
(dog, 2)
(fish, 2)
(cat, 3)
(fish, 3)

Shuffle Phase

Grouped by term
cat  → [1, 3]
dog  → [1, 2]
fish → [2, 3]

Reduce Phase

Final inverted index
cat  → [1, 3]
dog  → [1, 2]
fish → [2, 3]

Map emits (term, docID), Shuffle groups by term, Reduce builds postings lists.
